# 趁着最近面试，我重新梳理了一遍电商ERP

> **版本**：V1.0 | 2026-07-07

最近面试遇到好几家做电商ERP和OMS的公司，借此机会梳理下之前参与的电商ERP碎片化知识，尝试把一笔订单从平台到仓库的完整履约链路说清楚。

很多人第一反应是讲功能：订单审核、配货打单、称重发货、库存同步、售后处理。功能没错，但串起来看，电商ERP的核心其实可以概括成一句话：

**把平台上的交易订单，转换成仓库可以执行的发货任务。**

这篇文章就沿着这条主线，把一笔订单从平台到仓库的完整过程走一遍。

---

## 电商ERP到底是什么

名字叫ERP，但电商ERP跟传统ERP管的事情差别很大。它最核心的职责是连接——前连电商平台，后连仓库和WMS。

具体来说，它要做这几件事：

- 对接天猫、京东、抖音、拼多多、Amazon 等销售平台
- 统一接收多个平台、多个店铺的订单
- 完成商品识别、订单审核、寻仓锁库、拆合单和物流匹配
- 把发货任务下发给 WMS 或第三方仓
- 接收发货结果并回传平台

很多人容易把电商ERP和平台OMS、企业ERP、WMS搞混，因为大家都在管"订单"，但管的不是同一个东西：

| 系统 | 管的本质是什么 |
| --- | --- |
| 电商平台 | 交易是否成立、支付是否完成 |
| 电商ERP/OMS | 订单从哪里发、能不能发、怎么发 |
| 企业ERP | 商品档案、采购成本、财务核算 |
| WMS | 仓库里怎么拣、怎么打包、怎么出库 |

---

## 订单是怎么进来的

订单不是凭空出现在ERP里的。商家得先在ERP里完成店铺授权，ERP拿到订单读取和状态回传的权限后，平台才会把订单数据同步过来。

常见的同步方式有三种：

| 方式 | 特点 | 主要风险 |
| --- | --- | --- |
| 平台消息推送 | 时效高，平台有新订单后主动通知ERP | 消息可能丢失或重复 |
| 定时拉取 | ERP定期轮询，实现简单 | 接口限流、订单有延迟 |
| 手动拉取 | 运营按需补拉 | 可能重复，权限和时间范围要控制 |

真正考验产品设计的不是"拉取按钮放在哪"，而是：怎么防重复建单？平台修改订单后怎么同步？漏单了怎么发现和补偿？平台取消订单时ERP已经锁库了怎么办？多店铺之间如何做数据隔离？

---

## 第一道门槛：SKU映射

订单进来了，能不能直接发货？不行。第一个要解决的问题是：平台卖的那个商品，对应公司内部的哪个SKU？

平台有一套SKU编码，公司ERP有一套，WMS可能还有一套。三方仓的SKU编码又不一样。如果ERP不知道它们之间的对应关系，后面所有事情都做不了：不知道哪个仓库有货、不知道该锁哪份库存、拣货单上该写什么。

一个常见的错误做法是等订单来了再临时处理SKU映射。合理的方式应该在商品上架时就维护好映射关系。订单进来后做一次校验，映射缺失的直接打入异常池，不往下走。

---

## 订单审核：把交易数据变成可履约的任务

很多人的理解里，审核就是人工点一下"通过"。成熟电商ERP里的审核，本质上是一套规则引擎。

一笔订单从进来审核到通过，要经过的判断包括：

- SKU映射是否完整
- 收货地址是否有效
- 商品是否有库存
- 应该从哪个仓库发货
- 应该用哪家物流
- 是否需要加赠品
- 是否命中黑名单或风控规则
- 是否需要拆单、合单或挂起

这些判断有先后顺序。一个典型的执行链路是这样：

平台订单 → 基础校验 → SKU映射校验 → 合单/风险判断 → 库存查询 → 寻仓 → 锁库 → 拆单配物流 → 生成发货任务

这里有几个容易搞混的时序问题：合单通常发生在明确分仓之前——两个订单如果发同一个地址，可以先合并再决定从哪发。拆分通常发生在确仓的时候——一个订单的商品分布在多个仓库，按仓库拆成多个发货单。挂起可以在多个阶段触发，风控不合格、地址异常、库存不足都可能触发挂起。

---

## 寻仓和锁库，时序很重要

同一个商品可能同时存在多个仓库。系统需要决定从哪个仓库发，这就是寻仓。

判断条件包括：仓库有没有货、仓库是否覆盖收货地址所在区域、店铺是否允许使用这个仓库、物流时效和成本、是否需要跨仓拆单。

但查到有货不等于一定能发出去。查询库存和锁定库存是两回事。大促期间几百个订单同时进来，查询时有货，点击锁库时可能已经被其他订单占了。正确时序应该是：

1. 查询候选仓库的可用库存
2. 按规则选出最优仓库
3. 尝试锁定库存
4. 锁库成功 → 进入发货流程
5. 锁库失败 → 重新寻仓或进入缺货异常

锁库失败后的处理方案也要考虑：切换备选仓、拆成多个仓发货、等待补货、或者释放订单走人工处理。

---

## 本地仓和第三方仓，履约逻辑不一样

很多电商ERP同时对接自有WMS和第三方仓。对接第三方仓不只是换个API接口，本质上是把一部分库存真实性和履约控制权交给了外部系统。

| 对比项 | 自有WMS | 第三方仓 |
| --- | --- | --- |
| 库存掌控程度 | 内部系统，实时可控 | 依赖对外接口查询 |
| 发货指令 | 下发出库任务到WMS | 调用第三方仓API |
| 库存校验 | 可直接结合本地库存判断 | 需要再次查询三方仓库存 |
| 异常处理 | 内部系统可直接干预 | 等接口结果或人工协调 |
| 取消发货 | 未拣货前通常可拦截 | 三方仓受理后可能无法取消 |

一个常见的坑是：本地仓的库存管理逻辑不能直接套用到第三方仓上。第三方仓有自己的作业节奏和系统流程，ERP能做的是把指令发清楚、把结果接回来，中间的细节控制不了。

---

## 发出去以后，流程还没结束

仓库完成拣货、复核、打包、称重、获取物流单号、确认出库后，信息要沿着来路回流：

仓库出库 → WMS回传结果 → ERP更新发货状态 → 回传物流单号 → 平台更新为已发货

这个环节容易出的问题不少：仓库已经发货了但ERP没收到回传、ERP已经发货了但平台回传失败、一个订单拆成多个包裹、物流单号错了被更换、平台订单取消了但仓库已经出库、接口重试导致重复发货通知。每一个场景都需要做对应的异常处理设计。

---

## B2C电商和B2B履约，本质差异在哪

| 对比项 | B2B销售履约 | B2C电商履约 |
| --- | --- | --- |
| 订单来源 | 销售录单、客户订单、合同 | 电商平台自动同步 |
| 订单特点 | 数量少、金额高、流程长 | 高频、小单、时效要求高 |
| 核心关注点 | 价格、信用、账期、分批交付 | 自动审单、库存、物流、平台规则 |
| 库存处理 | 通常在审核时预占 | 多个履约节点都可能校验 |
| 履约方式 | 按客户计划组织发货 | 自动寻仓、拆单、配货 |
| 结果反馈 | 回写ERP和财务系统 | 必须及时回传平台 |
| 异常处理 | 更多依赖人工协调 | 依赖规则引擎和异常订单池 |

B2B强调的是业务控制——每一笔交易都有对应的客户、合同和信用审批。B2C强调的是规模化自动处理——一天几万单，不可能人工逐单审核。

---


回到开头那句话，把一笔订单走完：

平台接单 → 订单同步 → SKU映射 → 订单审核 → 寻仓锁库 → 生成发货任务 → 仓库出库 → 结果回传平台

以前我看电商ERP，注意力经常放在订单列表、审核按钮和各种状态标签上。这次重新梳理后，我更愿意把它理解成一个连接器和决策中心——前面连接各个电商平台，中间处理订单、库存和履约规则，后面对接WMS和第三方仓。

页面可以不同，产品名称可以不同，但主线不会变：**接住订单，找到库存，组织发货，再把结果准确送回平台。**
